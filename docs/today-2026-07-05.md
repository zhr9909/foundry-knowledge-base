# Progress Update (2026-07-05)

## 1. rank_bm25 → PostgreSQL tsvector 全文检索

### 动机
rank_bm25 在内存中加载 29,585 条 chunk 全量分词构建倒排索引，占用 ~150MB 内存，启动耗时 ~20s，
且新增文档必须重启服务。改用 PostgreSQL 自带的 tsvector + GIN 索引，索引存在磁盘上，无需预加载。

### 数据库变更

```sql
ALTER TABLE chunks ADD COLUMN fts tsvector
  GENERATED ALWAYS AS (to_tsvector('english', coalesce(content_text, ''))) STORED;

CREATE INDEX chunks_fts_idx ON chunks USING GIN(fts);

ANALYZE chunks;  -- 更新统计信息，让 ts_rank() 的 IDF 计算准确
```

已有 29,585 行自动回填 fts 列，后续 INSERT 增量维护。

### 代码变更

| 变更 | 文件 | 说明 |
|------|------|------|
| 删除 `_init_bm25()` | search.py | 移除全量加载 29,585 chunk 到内存的逻辑 |
| 删除 `_bm25_search()` | search.py | 移除 rank_bm25 打分 + 排序 |
| 删除 `from rank_bm25 import BM25Okapi` | search.py | 不再需要该库 |
| 删除 `from collections import Counter` | search.py | 不再需要 |
| 新增 `_tsvector_search()` | search.py | 调用 PostgreSQL plainto_tsquery + ts_rank |
| 更新调用 | search.py | `bm25_res = _bm25_search(...)` → `_tsvector_search(...)` |
| 更新 warmup() | search.py | 不再 preload BM25，只有 embedding 模型 |
| 更新启动消息 | search.py | "model + BM25" → "embedding model + PostgreSQL tsvector" |

### 性能对比

| 指标 | rank_bm25（之前） | tsvector（现在） |
|------|-------------------|------------------|
| 启动耗时 | ~20 秒（全量加载分词） | ~0 秒 |
| Python 进程内存 | ~150 MB | 0 |
| 新增文档后 | 必须重启服务 | INSERT + ANALYZE 即可 |
| 查询速度 | 毫秒级 | 毫秒级（使用 GIN 索引） |
| 查询计划 | - | `Bitmap Index Scan on chunks_fts_idx` |

### tsvector 查询核心函数

```python
def _tsvector_search(query, top_k=40, cur=None):
    sql = """
        SELECT c.id, c.chunk_id,
            ts_rank(c.fts, plainto_tsquery('english', %(q)s)) AS score
        FROM chunks c
        WHERE c.fts @@ plainto_tsquery('english', %(q)s)
        ORDER BY score DESC LIMIT %(limit)s
    """
    cur.execute(sql, {"q": query, "limit": top_k})
    ...
```

## 3. RRF 融合评分详细日志

### 改动
在 `_rrf_fuse()` 函数中添加详细评分分解日志，输出到 `agent.log`：

```python
_sqllog.info(f"=== RRF Scoring Breakdown (k={k}, top={top_k}) ===")
for i, r in enumerate(top):
    vr = r.get("vr", 999); br = r.get("br", 999)
    term_v = 1.0 / (k + vr); term_b = 1.0 / (k + br)
    calc_rrf = round((term_v + term_b) / 2.0, 4)
    _sqllog.info(
        f"  [{i+1}] VR={vr} -> 1/(60+{vr})={term_v:.6f} | "
        f"BR={br} -> 1/(60+{br})={term_b:.6f} | "
        f"RRF=({term_v:.6f}+{term_b:.6f})/2={calc_rrf:.4f}"
    )
```

### 修复 `vr`/`br` 未写入问题
`git checkout` 还原文件导致用户原来写的 `d["vr"] = vr / d["br"] = br` 丢失，
日志中所有条目显示 VR=999 / BR=999。已重新补回 `_rrf_fuse()` 两个分支中。

### 日志示例

## 4. 修复 `search()` 中 `else` 分支（非 hybrid 模式）

### 问题
`search()` 的 `else` 分支（当 `hybrid=False` 时）：
- 只定义了 SQL 字符串和 params，从未执行 `cur.execute(sql, params)`
- 未赋值 `results` → 后续代码用到时 NameError
- 用户新增的 `bm25_raw_json` 仅定义在 `if hybrid:` 分支中

### 修复
补上 `cur.execute(sql, params)`、`rows = cur.fetchall()`、`results` 构建循环，
以及 `bm25_raw_json = "[]"`。
