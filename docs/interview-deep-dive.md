## 一、架构类

### Q1: 为什么选 pgvector 而不是 Pinecone、Weaviate 这些专用向量数据库？

> ❌ 差回答："因为 pgvector 免费"

✅ **好回答：**

"三个原因：

第一，**事务一致性。** 我的业务数据（合同原文、章节路径）和向量数据存在同一张表里，不需要跨系统维护一致性。Pinecone 存向量，PostgreSQL 存原文——更新原文时向量不同步怎么办？pgvector 用同一个 `INSERT` 就搞定了。

第二，**减少运维组件。** 我现在 PostgreSQL 已经跑在 Docker 里，加 pgvector 就是一行 `CREATE EXTENSION`。不用额外管理一个向量数据库集群。

第三，**SQL JOIN。** 我可以把向量检索和章节目录过滤写在一个 `WHERE` 条件里，Pinecone 需要先向量检索拿到 ID 列表再回 PostgreSQL 查原文——多一次网络往返。"

### Q2: 你的检索架构和 LangChain 的 RetrievalQA 有什么区别？

> ❌ 差回答："没用 LangChain，我是自己写的。"

✅ **好回答：**

"我选择了**手撸编排**而不是 LangChain，原因是对标 AI Solution Architect 的能力要求：

- **审计需求**：金融/制造行业的合规要求每一步有迹可查。LangChain 的 chain 内部调用链是黑盒，我手撸的编排每个步骤（解析→检索→工具调用→LLM 生成→人工复核）都有显式的日志和状态管理。
- **面试展示深度**：手撸 RRF 融合算法、手写 metadata 回填、自己设计去重逻辑——这些比"调 LangChain 的 RetrievalQA"更能证明架构能力。
- **不过度抽象**：当需要调整检索策略时，我直接改几行 SQL，不需要绕 LangChain 的 abstraction layer。"

### Q3: 如果数据量从 5000 条增长到 500 万条，你的架构怎么扩展？

✅ **好回答：**

"分三步：

**第一，索引升级。** 现在的 IVFFlat 索引换成 HNSW（pgvector 0.7+ 支持），召回率不降的情况下查询速度从秒级降到毫秒级。

**第二，分区表。** `chunks` 表按 `source_id` 做 PostgreSQL 分区，不同文档源分到不同的物理段。

**第三，分区检索 + 分布式。** 如果单机扛不住，引入 pgvector 的流式副本（Patroni + pgvector），只读流量打到副本，写入主库。

不过对于 5000 条这个量级，这些优化都不需要。50ms 已经够了。"

## 二、数据类

### Q4: PDF 表格提取失败怎么办？数据质量怎么保证？

✅ **好回答：**

"我的策略是**降级保留，关键数据人肉确认**。

技术层面：pdfplumber 的 `extract_text()` 包含了表格内容的行内重排，虽然表结构乱了，但数据（"Si=0.25%, Fe=0.30%"）没有丢。对 RAG 检索来说，**语义内容保留比结构保留更重要**——用户搜"硅含量"能命中就行，不需要看到原表。

质量控制：我建了 5 份测试文档（含标准表格、复杂合并单元格表格、纯文本章节），定义了期望的检索命中率。每次改清洗逻辑都跑一遍。

对于高频使用的关键文档（比如你爸的铸造工艺手册），人工过一遍提取结果，标记需要修复的表格。"

### Q5: metadata 为什么不用 LLM 提取？

✅ **好回答：**

"两句话：**PDF 原生结构比 LLM 更可靠、更便宜。**

PDF 大纲（/Outlines）是 PDF 规范的一部分，出版社自己在 Acrobat 里画的章节层级，页码绝对精确。LLM 猜测章节名可能幻觉（把"铝合金"猜成"铝"或"Aluminum Alloys"）。

不过 LLM 有它的定位——**当 PDF 没有大纲结构时**（扫描件、老旧 PDF），才用 LLM 做章节推演和实体提取。原则是：硬结构优先，AI 兜底。"

## 三、检索类

### Q6: 你的 RRF 混合检索参数 k 为什么取 60？调过参吗？

✅ **好回答：**

"标准 RRF 论文（Cormack et al. 2009）建议的 k=60，是一个经验上对大多数场景鲁棒的默认值。我的数据集只有 5000 条，做了简单的 ablation：k=60 vs k=30 vs k=100，结果差异不大。如果数据量增大，我会做一个 `nDCG@10` 的网格搜索来选最优 k。"

### Q7: 用户搜中文关键词（如"抗拉强度"），你的英文文档能命中吗？

✅ **好回答：**

"两个层面都能命中：

第一，**向量检索**。bge-small-zh-v1.5 是中英双语跨语言模型——即使 chunk 里全是英文，"抗拉强度"的向量和"tensile strength"的向量在语义空间里是接近的。实测 73ms 找到 6061 的力学性能表。

第二，**全文检索**。目前用的是 `english` 分词器，中文确实不适用。但我的下一版计划是安装 `pg_jieba` 扩展，支持中文分词。当用户把铸造行业的中文手册加进来时，全文检索也能命中。问题有预期解决方案。"

### Q9: 换一份新的铸造 PDF，接入流程有多长？

✅ **好回答：**

"三条命令：

```bash
python scripts/pdf_extractor.py raw/新手册.pdf processed/
python scripts/ingest.py processed/新手册_extracted_*.jsonl
# backfill_metadata 自动检测新文档的 TOC
python scripts/backfill_metadata.py raw/新手册.pdf
```

如果新 PDF 有书签目录，整个流程不需要改代码。如果新 PDF 是扫描件（没有可提取文本），需要先 OCR——这个后续可以加。"

## 五、追问类（追问越深，面试官越认可）

### Q10: 如何评估你的 RAG 质量？

✅ **好回答：**

"目前是**人工评估**——对 5 个测试查询观察 top-3 结果的相关性。

后续计划建立量化评估：
- **检索层**：Hit Rate（期望命中是否在前 5）、MRR（排第几位找到的）
- **生成层**（如果接 LLM）：Faithfulness（回答是否基于检索结果）、Answer Relevance（回答是否解决问题）

5 份合成测试合同我已经准备好了，后续跑一个脚本就能出评估报告。"

### Q11: 如果你们工厂的师傅想用语音问问题，前端怎么设计？

✅ **好回答：**

"前端加一个语音输入按钮（Web Speech API），语音识别转文本 → 调用 `/search` 接口 → 返回结果朗读（TTS）。后端不需要改一行代码。"

## 六、你可以主动聊的亮点

面试中找机会自然引出这些：

1. **"Source Structure First" 原则** → metadata 来自 PDF 目录而非 LLM，展示架构判断力
2. **RRF 混合检索** → 聊你对比过纯向量和混合的差异，展示工程思维
3. **pgvector 选型** → 和 Pinecone/Chroma 对比过，展示技术广度
4. **清洗去重策略** → 你意识到了表格和正文的重复问题并解决了它
5. **零 token 成本提取** → 没用 LLM 洗数据，展示成本意识

### Q12: 你的 RAG 系统从 V1 到 V2 做了什么改进？

✅ **好回答：**

"V1 的问题是 RRF 公式把分数全抹平了——无论好坏文档都显示 0.016 分，LLM 根本分不清哪些是高质量的上下文。中文查询的第一条结果是铜合金，完全答非所问。

V2 做了三件事：

第一，**修复打分**。把 `1/(60+rank)` 换成了 `向量相似度 + 全文命中加分`，分数回到 0.48-0.70 的正常区间。排名立刻合理了，6061 的力学性能表排到了第一。

第二，**查询改写**。用户输入中文"铝合金6061的力学性能"，先用 DeepSeek 转成英文搜索词 `["6061 aluminum alloy mechanical properties", "6061-T6 tensile strength"]`。英文文档用英文搜，准确率更高。

第三，**并行搜索**。如果改写出来多个子查询（比如"对比6061和7075"拆成两个），用 ThreadPoolExecutor 并行搜，结果汇总去重。不增加总耗时，但覆盖更广。"

### Q13: 为什么不用 LLM 做整个 Agent 编排，而是混合了规则 + LLM？

✅ **好回答：**

"成本与速度的权衡。

查询改写用了 LLM（2-3 秒，约 50 token），因为语义理解确实是 LLM 的强项——它能理解 '力学性能' 对应 'mechanical properties'。

但检索和排序用的是规则（向量搜索 + FTS + 分数归一化），因为：
- **确定性**：检索结果不应该有随机性，同样的查询每次应该返回同样结果
- **速度**：检索在 50ms 内完成，如果交由 LLM '思考' 每一步，延迟会从秒级变成分级
- **可审计**：每一步都有日志，可以回溯为什么某条文档被选中或被忽略

我的原则是：**LLM 做理解（查询改写），规则做执行（检索、打分、排序）**。这样兼得两者的优势。"
