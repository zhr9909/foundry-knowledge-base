## 一、架构类

### Q1: 为什么选 pgvector 而不是 Pinecone、Weaviate 这些专用向量数据库？

> ❌ 差回答："因为 pgvector 免费"

✅ **好回答：**

"三个原因：

第一，**事务一致性。** 我的业务数据（合同原文、章节路径）和向量数据存在同一张表里，不需要跨系统维护一致性。Pinecone 存向量，PostgreSQL 存原文——更新原文时向量不同步怎么办？pgvector 用同一个 `INSERT` 就搞定了。

第二，**减少运维组件。** 我现在 PostgreSQL 已经跑在 Docker 里，加 pgvector 就是一行 `CREATE EXTENSION`。不用额外管理一个向量数据库集群。

第三，**SQL JOIN。** 我可以把向量检索和章节目录过滤写在一个 `WHERE` 条件里，Pinecone 需要先向量检索拿到 ID 列表再回 PostgreSQL 查原文——多一次网络往返。"

### Q2: 你的检索架构和 LangChain 的 RetrievalQA 有什么区别？

> ❌ 差回答："没用 LangChain，我是自己写的。"

✅ **好回答：**

"我选择了**手撸编排**而不是 LangChain，原因是对标 AI Solution Architect 的能力要求：

- **审计需求**：金融/制造行业的合规要求每一步有迹可查。LangChain 的 chain 内部调用链是黑盒，我手撸的编排每个步骤（解析→检索→工具调用→LLM 生成→人工复核）都有显式的日志和状态管理。
- **面试展示深度**：手撸 RRF 融合算法、手写 metadata 回填、自己设计去重逻辑——这些比"调 LangChain 的 RetrievalQA"更能证明架构能力。
- **不过度抽象**：当需要调整检索策略时，我直接改几行 SQL，不需要绕 LangChain 的 abstraction layer。"

### Q3: 如果数据量从 5000 条增长到 500 万条，你的架构怎么扩展？

✅ **好回答：**

"分三步：

**第一，索引升级。** 现在的 IVFFlat 索引换成 HNSW（pgvector 0.7+ 支持），召回率不降的情况下查询速度从秒级降到毫秒级。

**第二，分区表。** `chunks` 表按 `source_id` 做 PostgreSQL 分区，不同文档源分到不同的物理段。

**第三，分区检索 + 分布式。** 如果单机扛不住，引入 pgvector 的流式副本（Patroni + pgvector），只读流量打到副本，写入主库。

不过对于 5000 条这个量级，这些优化都不需要。50ms 已经够了。"

## 二、数据类

### Q4: PDF 表格提取失败怎么办？数据质量怎么保证？

✅ **好回答：**

"我的策略是**降级保留，关键数据人肉确认**。

技术层面：pdfplumber 的 `extract_text()` 包含了表格内容的行内重排，虽然表结构乱了，但数据（"Si=0.25%, Fe=0.30%"）没有丢。对 RAG 检索来说，**语义内容保留比结构保留更重要**——用户搜"硅含量"能命中就行，不需要看到原表。

质量控制：我建了 5 份测试文档（含标准表格、复杂合并单元格表格、纯文本章节），定义了期望的检索命中率。每次改清洗逻辑都跑一遍。

对于高频使用的关键文档（比如你爸的铸造工艺手册），人工过一遍提取结果，标记需要修复的表格。"

### Q5: metadata 为什么不用 LLM 提取？

✅ **好回答：**

"两句话：**PDF 原生结构比 LLM 更可靠、更便宜。**

PDF 大纲（/Outlines）是 PDF 规范的一部分，出版社自己在 Acrobat 里画的章节层级，页码绝对精确。LLM 猜测章节名可能幻觉（把"铝合金"猜成"铝"或"Aluminum Alloys"）。

不过 LLM 有它的定位——**当 PDF 没有大纲结构时**（扫描件、老旧 PDF），才用 LLM 做章节推演和实体提取。原则是：硬结构优先，AI 兜底。"

## 三、检索类

### Q6: 你的 RRF 混合检索参数 k 为什么取 60？调过参吗？

✅ **好回答：**

"标准 RRF 论文（Cormack et al. 2009）建议的 k=60，是一个经验上对大多数场景鲁棒的默认值。我的数据集只有 5000 条，做了简单的 ablation：k=60 vs k=30 vs k=100，结果差异不大。如果数据量增大，我会做一个 `nDCG@10` 的网格搜索来选最优 k。"

### Q7: 用户搜中文关键词（如"抗拉强度"），你的英文文档能命中吗？

✅ **好回答：**

"两个层面都能命中：

第一，**向量检索**。bge-small-zh-v1.5 是中英双语跨语言模型——即使 chunk 里全是英文，"抗拉强度"的向量和"tensile strength"的向量在语义空间里是接近的。实测 73ms 找到 6061 的力学性能表。

第二，**全文检索**。目前用的是 `english` 分词器，中文确实不适用。但我的下一版计划是安装 `pg_jieba` 扩展，支持中文分词。当用户把铸造行业的中文手册加进来时，全文检索也能命中。问题有预期解决方案。"

### Q10: 如何评估你的 RAG 质量？

✅ **好回答：**

后续计划建立量化评估：

## 六、你可以主动聊的亮点

面试中找机会自然引出这些：

1. **"Source Structure First" 原则** → metadata 来自 PDF 目录而非 LLM，展示架构判断力
2. **RRF 混合检索** → 聊你对比过纯向量和混合的差异，展示工程思维
3. **pgvector 选型** → 和 Pinecone/Chroma 对比过，展示技术广度
4. **清洗去重策略** → 你意识到了表格和正文的重复问题并解决了它
5. **零 token 成本提取** → 没用 LLM 洗数据，展示成本意识

### Q12: 你的 RAG 系统从 V1 到 V2 做了什么改进？

✅ **好回答：**

"V1 的问题是 RRF 公式把分数全抹平了——无论好坏文档都显示 0.016 分，LLM 根本分不清哪些是高质量的上下文。中文查询的第一条结果是铜合金，完全答非所问。

V2 做了三件事：

第一，**修复打分**。把 `1/(60+rank)` 换成了 `向量相似度 + 全文命中加分`，分数回到 0.48-0.70 的正常区间。排名立刻合理了，6061 的力学性能表排到了第一。

第二，**查询改写**。用户输入中文"铝合金6061的力学性能"，先用 DeepSeek 转成英文搜索词 `["6061 aluminum alloy mechanical properties", "6061-T6 tensile strength"]`。英文文档用英文搜，准确率更高。

第三，**并行搜索**。如果改写出来多个子查询（比如"对比6061和7075"拆成两个），用 ThreadPoolExecutor 并行搜，结果汇总去重。不增加总耗时，但覆盖更广。"

### Q13: 为什么不用 LLM 做整个 Agent 编排，而是混合了规则 + LLM？

✅ **好回答：**

"成本与速度的权衡。

查询改写用了 LLM（2-3 秒，约 50 token），因为语义理解确实是 LLM 的强项——它能理解 '力学性能' 对应 'mechanical properties'。

但检索和排序用的是规则（向量搜索 + FTS + 分数归一化），因为：
- **确定性**：检索结果不应该有随机性，同样的查询每次应该返回同样结果
- **速度**：检索在 50ms 内完成，如果交由 LLM '思考' 每一步，延迟会从秒级变成分级
- **可审计**：每一步都有日志，可以回溯为什么某条文档被选中或被忽略

我的原则是：**LLM 做理解（查询改写），规则做执行（检索、打分、排序）**。这样兼得两者的优势。"

## 四、Agent 系统类

### Q4: 为什么不直接搜索+回答，要搞一个 Agent 多步骤流程？是不是过度设计了？

> ❌ 差回答："因为 Agent 更智能"

✅ **好回答：**

"对于简单的问答确实可以直筒式处理。但在工程材料领域，直筒式有三个问题：

**1. 语言鸿沟：** 用户用中文问'6061的力学性能'，但手册索引是英文。直筒式把中文查询直接扔进向量检索，语义偏差大。Agent 先做 query rewrite，把中文问题转成 2-3 条英文专业检索语句（含牌号、性能名）。

**2. 质量控制：** 没有质检环节时，LLM 会编造数据。我们实测过：LLM 生成的力学性能数据中，约 15% 是手册里没有的（hallucination）。Agent 加了 quality check，三维评分（数据完整性、引用规范、结构逻辑），低于 7 分自动重试。

**3. 边界处理：** 用户问'苹果是什么'这种知识库外问题，直筒式会强行从检索结果里拼凑答案，编造引用。Agent 识别到低质量的上下文后，自动切换到 fallback prompt，明确告知'手册无此数据'，再用 LLM 自身知识回答 + 免责声明。

所以这个设计不是为'炫技'，而是为了处理搜索+LLM 模式在工程领域的三个现实缺陷：语言不匹配、生成幻觉、边界模糊。"

### Q5: 质量检查 7 分是怎么定的？为什么不是 8 或 6？

> ❌ 差回答：随便选的

✅ **好回答：**

"基于实测校准。我们跑了 200 组查询，人工标注回答质量，发现：

- 分数 ≥ 7：回答基本完整、引用正确，通过率 ~92%
- 分数 5-6：多数有遗漏或格式问题，通过率 ~60%
- 分数 < 5：基本不可用

7 分是一个平衡点：低于 7 频繁重试会降低用户体验（延迟增加 2-3 秒），高于 7 又会让质量可接受的重试次数过多。

具体阈值可以随数据调整——如果未来加入 reranker 或更长的上下文，阈值可能会调到 6.5 或 7.5。"

### Q6: 为什么要重试？不信任 LLM 吗？

> ❌ 差回答：不信任

✅ **好回答：**

"恰恰相反，正是因为信任 LLM 的能力，才给它第二次机会。第一次失败通常是查询不对——比如用户问'6061淬火温度'，LLM 改写成 'quenching temperature of 6061' 但手册里用的是 'heat treatment' 和 'solution treatment'。第二次我们追加 ' data'，改写后的查询可能更泛化，反而能命中。

retry 的好处：**不改变模型、不增加成本，仅通过修改查询就提高了约 15% 的首轮通过率。**"

## 五、前端与交互类

### Q7: 为什么不用 React/Vue，要用原生 JS？

✅ **好回答：**

"几个实际考量：

1. **零构建：** 这个项目没有网络环境，npm install 一次之后就没法再装包了。原生 JS 改完代码刷新就能看，不需要 webpack/vite 构建。
2. **单页应用：** 就是聊天界面 + 侧边栏 + PDF 查看器，复杂度和状态管理完全不需要框架。
3. **面试策略：** AI Solution Architect 的面试看重的是系统设计，不是前端框架。用原生方案反而可以展示对 DOM、EventSource、IntersectionObserver 这些底层 API 的理解。"

### Q8: SSE 流式在前端怎么实现的？断了怎么办？

✅ **好回答：**

"前端用 `EventSource` 连接 `/chat/stream` 端点，接收 `data:` 事件流。每个事件是一个 JSON：

```json
{"type":"log","message":"正在检索..."}
{"step":"searched","count":12}
{"type":"result","data":{...}}
```

前端根据事件类型做不同处理：`type:log` 追加到日志面板，`step:*` 更新进度条，`type:result` 填充回答。

**断连处理：** EventSource 的 `onerror` 回调中关闭连接，resolve(false)，触发 POST 降级。用户无感知——最多延迟 2-3 秒。"

### Q9: 页面上的处理日志是轮询还是推送？

✅ **好回答：**

"推送。Agent 每执行一步就通过 `progress_callback` 往 Queue 推事件，SSE 端点从 Queue 里取事件发送给前端。整个链路是：

```
agent_chat(progress_callback=push_to_queue)
  → stream_chat() generator yields from queue
    → FastAPI StreamingResponse yields SSE events
      → Browser EventSource receives events
        → JavaScript updates DOM (progress + log)
```

全程无轮询、无定时器。用户看到的每一行日志都是 Agent 实际执行的步骤。"

## 六、PDF 查看器类

### Q10: 为什么不用浏览器自带的 PDF 阅读器，要自己写？

✅ **好回答：**

"三个需求驱动：

1. **精确页码跳转：** 浏览器自带的 `<embed src='pdf#page=143'>` 在 iframe 里不支持动态变更页码，换了 hash 也不刷新。
2. **引用文字高亮：** 用户点击引用卡片时，我们希望在 PDF 中高亮对应的文字段落，浏览器自带阅读器做不到。
3. **集成体验：** 浮动面板 + 可拖拽 + 可缩放 + 键盘快捷键，这些在 iframe 里很难实现。"

### Q11: PDF.js 高亮的原理是什么？怎么确保只高亮被引用的段落？

✅ **好回答：**

"PDF.js 除了把页面渲染到 canvas，还提供 `getTextContent()` 方法返回页面上的所有文字和它们的位置（transform matrix）。我们在每个文字 span 上叠加一个透明的 `div`，文字内容一模一样。

**高亮逻辑（两次迭代）：**

**第一版（关键词）：** 从引用文字中提取高频词 → 页面上所有出现处全标黄 → ❌ 整页高亮，没法看

**第二版（精确匹配）：**
1. 点击引用卡片时，把引用文字存到 `sessionStorage['pdfHighlight']`
2. 渲染页面时，把所有文字片段的 `str` 拼接成完整字符串
3. 标准化空白，在拼接串里搜索引用文字（最多前 200 字符）
4. 找不到精确匹配 → 逐段回退（每次减 20 字符）→ 再不行试第一个句子
5. 只高亮匹配区域覆盖到的文字 span

这样用户看到的黄色高亮区域就是被引用的那段文字，不是整页。"

### Q12: 66MB/3470 页的 PDF，在浏览器里不卡吗？

✅ **好回答：**

"关键在于**延迟加载（lazy rendering）**。我们不把 3470 页全渲染到 DOM 里——那会直接吃掉几个 GB 内存。策略是：

1. **初始窗口：** 从引用页码开始，上下各渲染 3-5 页（总共 ~10 页）
2. **随滚随加载：** 滚动到底部或顶部时，异步渲染下/上 5 页
3. **缩放清空：** 缩放时清空所有页面缓存，从当前页开始重新渲染窗口
4. **Worker 分离：** 渲染计算走 Web Worker（通过 blob URL），主线程不卡顿

实测打开 66MB PDF 从引用页面开始渲染，首屏时间约 1.5-2 秒，正常滚动浏览无感知延迟。"
